## Executive summary

Nemovizor MVP je **funkční, technicky zralá platforma** s dobrou základní architekturou (Next.js 16, Supabase, Cloudflare R2, Stripe). API surface je velmi dobře navržené pro AI agenty (MCP, OpenAPI 3.1, Zod schemas, audit log).

**ALE:** Audit odhalil **3 kritické security díry** a desítky drobnějších problémů napříč celým systémem. Před produkcí je nutné minimálně:
1. Vrátit RLS po migraci 004 (otevřená DB)
2. Přidat Stripe signature verification do subscription webhook
3. Doplnit GDPR consent na lead formuláře

## 1. KRITICKÉ ZÁVADY (musí se opravit před produkcí)

### 1.1 🔴 RLS je globálně otevřené (migrace 004)
**Soubor:** `supabase/migrations/004_relax_rls_mvp.sql`
**Dopad:** Anonymní uživatel může INSERT/UPDATE/DELETE jakoukoliv property, broker, agency, branch, review přes anon key. Komentář říká "TODO: zpřísnit zpět" — nikdy nezpřísněno.
**Oprava:** Nová migrace která vrátí restriktivní RLS. Tabulky které potřebují anon WRITE (jen `properties` přes scrapery, `leads` přes lead form) musí mít explicit policies. Vše ostatní musí jít přes `service_role`.

### 1.2 🔴 Stripe subscription webhook neověřuje podpis
**Soubor:** `src/app/api/subscriptions/webhook/route.ts`
**Dopad:** Útočník může POSTnout fake `customer.subscription.created` event s libovolným user_id → automaticky upgradne API klíč na vyšší tier (rate limit 10000/min, write:webhooks scope).
**Oprava:** Přidat `stripe.webhooks.constructEvent(body, sig, STRIPE_SUBSCRIPTION_WEBHOOK_SECRET)` jak to dělá wallet webhook.

### 1.3 🔴 Lead formuláře bez GDPR consent
**Soubory:** `src/app/oceneni/page.tsx`, `src/app/prodat/page.tsx`, kontaktní formuláře
**Dopad:** Sběr osobních dat bez explicitního souhlasu = porušení GDPR čl. 6.
**Oprava:** Povinný checkbox "Souhlasím se zpracováním osobních údajů" + odkaz na privacy policy.

### 1.4 🔴 Cover_photo a 30+ polí se neukládala (FIXED)
**Soubor:** `src/app/api/admin/brokers/route.ts`, `agencies/route.ts`
**Status:** ✅ Opraveno v commit `1bff81b`. Whitelist BROKER_FIELDS a AGENCY_FIELDS rozšířen.
**Pozn:** AGENCY_FIELDS stále nemá `awards` field — nutno doplnit.

## 6. PERFORMANCE

### 6.1 Offset pagination na 73k properties — 🟠 HIGH

**Soubor:** `src/lib/api/properties-data.ts`
- Hluboké stránkování (page 100+) skenuje 100k řádků
- Cursor pagination máme implementovanou (`base64url(created_at, id)`) ale **default je stále page-based**
- AI agenti používají page-based → pomalé

### 6.2 force-dynamic na všech API — 🟠 HIGH

**Většina `src/app/api/*/route.ts` má `export const dynamic = "force-dynamic"`**
- Disablovává Vercel edge cache
- Zvyšuje latency pro frequent reads (filter-options, properties list)
- Doporučení: přidat `revalidate: 60` na read-only GET endpointy

### 6.3 Chybějící DB indexy — 🟡 MEDIUM

- `brokers(agency_id)` — používá se ve filter-options RPC, žádný index
- `api_request_log(api_key_id)` — pro audit dashboard
- `api_request_log(client_hash)` — pro per-IP analytics

### 6.4 Playwright/Puppeteer v main deps — 🟡 MEDIUM

**Soubor:** `package.json`
- `playwright-extra`, `puppeteer-extra-plugin-stealth` jsou v `dependencies`
- Měly by být v `devDependencies` (jen scrapery je používají)
- Přidávají ~30MB k Vercel buildu

### 6.5 No image lazy-loading consistency — 🟡 MEDIUM

- PropertyCard používá Next.js Image s `loading="lazy"` ✓
- Broker/agency avatary používají plain `<img>` bez lazy loading
- Hero map dynamic import ✓

### 6.6 Listings-content monolit — 🟡 MEDIUM

1390 řádků, 58+ hooks. Re-render risk + bundle size impact.

### 6.7 No error monitoring — 🟡 MEDIUM

- Žádné Sentry / DataDog
- Cron jobs failures se ztratí pokud nikdo nečte Vercel logs
- Scrapers errors v console only

### 6.8 Webhook outbox bez TTL — 🟢 LOW

- `webhook_outbox` GC řešen cronem ale žádný hard TTL / partition strategy
- U vysokého traffic by mohl narůst

### 6.9 No request deduplication — 🟢 LOW

- Žádný React `cache()` wrapping na getterech
- Stejný getAgencyById může být volán 2x v jedné stránce
