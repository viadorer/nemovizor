**3. CRM Import (PTF.cz):**
```
PTF Worker → POST /api/v1/import/batch (s API klíčem write:import scope)
       ↓

       ↓ (return 202 + job_id)
Vercel cron */1 * * * * → /api/cron/process-imports
       ↓
Process 20 items/tick: agency → branches → brokers → properties
       ↓
Per-item: dedup (external_id+source) → upsert → image download → R2 upload
       ↓
Job complete → callback_url POST (HMAC signed)
       ↓
PTF poll /api/v1/import/jobs/{id} → update local nemovizor_synced_at
```

**4. AI agent (Claude Desktop přes MCP):**
```
User: "najdi byty 2+kk v Praze do 5M"
       ↓
Claude → mcp.nemovizor_ai_search({ query }) → Gemini parses → filters
       ↓
Claude → mcp.nemovizor_search_properties({ ...filters }) → camelCase results
       ↓
User: "kontakt na makléře u Property X"
       ↓
Claude → mcp.nemovizor_broker_contact({ id }) → rate-limited PII access
```

**5. Webhook delivery:**
```
Property INSERT/UPDATE/DELETE → Postgres trigger → webhook_outbox INSERT
       ↓
Vercel cron */5 * * * * → /api/cron/dispatch-webhooks
       ↓
Match subscriptions by event_type + filter → HMAC sign payload → POST URL
       ↓
Track delivery → exponential retry on failure → auto-disable po 20 selháních
```

## 2. SLABÁ MÍSTA — strukturovaná analýza

### 2.1 ARCHITEKTONICKÁ SLABÁ MÍSTA

#### **A. Žádná separace business logiky od HTTP vrstvy**
Každý route handler obsahuje validaci + auth + business logiku + DB operace. Při rozšíření na další frontend (mobilní app, white-label portál pro RK) bude třeba duplikovat.

**Řešení:** Service layer pattern:
```
src/services/property-service.ts  (čistá business logika)
src/app/api/v1/properties/route.ts (HTTP wrapper, deleguje)
```

#### **B. Žádný event bus / domain events**
Když se property změní, aktualizace musí být ručně volány na všech místech (webhook trigger, search index, analytics). Postgres triggery existují pro webhooks ale ne pro vše.

**Řešení:** Centralized event emitter v service layer:
```
propertyService.update(id, data)
  → DB update
  → emit('property.updated', { id, before, after })
  → listeners: webhook outbox, search reindex, analytics, audit
```

#### **C. RLS je vypnuté, security je app-level**
Migrace 004 globálně otevřela DB. Veškerá ochrana je v Next.js handlerech. **Pokud kdokoliv obejde Next.js (přímo na Supabase přes anon key) má plný read/write přístup.**

**Řešení:** Vrátit RLS, použít service_role client v Next.js handlerech.

#### **D. Monolitický listings-content (1390 řádků)**
Všechna logika filtru + mapy + listu + paginace v jednom client komponentu. Hard to test, hard to maintain.

**Řešení:** Split na 4 komponenty + shared FilterContext.

#### **E. Žádné separated concerns mezi v1 a legacy /api/***
Některé v1 endpointy proxy nad legacy, některé jsou direct handlers. Inkonzistence v PII filtering, audit log coverage.

**Řešení:** Sjednotit pattern. Buď vše v1Proxy(), nebo vše direct.

## 3. JAK S TÍM PRACOVAT — workflow doporučení

### 3.1 Pro tebe (David) — denní práce

**Když přidáváš novou feature:**
1. Začni v Plan mode — promysli architekturu, závislosti
2. Implementuj backend (route + Zod schema + audit log)
3. Implementuj UI (server components prioritně, client jen kde nutné)
4. Přidej do OpenAPI registry pokud public API
5. Přidej MCP tool pokud agent-relevant
6. `npx tsc --noEmit` před commitem
7. Push → Vercel deploy → manual smoke test

**Když fixuješ bug:**
1. Reprodukuj v dev / preview
2. Najdi root cause (ne jen symptom)
3. Napiš test pokud možno (přidá se do regresí)
4. Fix + commit s descriptive message
5. Push → Vercel deploy → ověř fix

**Když optimalizuješ:**
1. Změř baseline (curl s `-w "%{time_total}\n"`)
2. Identifikuj bottleneck (DB? bundle? rendering?)
3. Implementuj fix s graceful fallback
4. Změř po (musí být lepší, ne jen "možná")

### 3.2 Pro AI agenty (Claude) — jak používat Nemovizor

**Discovery flow:**
1. Stáhni `/llms.txt` (přehled API)
2. Stáhni `/api/openapi` (kompletní schema)
3. Identifikuj relevantní endpoint
4. Volej s API klíčem (Bearer) pro vyšší rate limity

**Search flow:**
1. POST `/api/ai-search` s natural language query → struktur. filtry
2. GET `/api/v1/properties` s filtry → výsledky
3. GET `/api/v1/properties/{id}` pro detail
4. GET `/api/v1/brokers/{id}/contact` jen pokud user explicitly chce kontakt

**Import flow (CRM):**
1. POST `/api/v1/import/batch` s entitami → 202 + job_id
2. GET `/api/v1/import/jobs/{id}` poll dokud `status === "completed"`
3. Zpracovat per-item results (warnings, errors, nemovizor_id)

### 3.3 Pro PTF.cz workera — concrete next steps

**Sprint 1.3 z PTF plánu (BullMQ Nemovizor worker):**
```typescript
// PTF backend: backend/src/workers/nemovizor.worker.ts
import { Worker } from 'bullmq';
import { redis } from '../services/redis';

new Worker('nemovizor-sync', async (job) => {
  const { tenantId, properties } = job.data;

  const res = await fetch('https://nemovizor.vercel.app/api/v1/import/batch', {
    method: 'POST',
    headers: {
      'Authorization': `Bearer ${process.env.NEMOVIZOR_API_KEY}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      external_source: `ptf-${tenantId}`,
      properties: properties.map(mapPropertyToNemovizor),
      callback_url: `https://ptf.cz/api/nemovizor-callback`,
    }),
  });

  const { job_id } = await res.json();

  // Poll for completion
  let completed = false;
  while (!completed) {
    await new Promise(r => setTimeout(r, 30_000));
    const status = await fetchJobStatus(job_id);
    if (status.status === 'completed') {
      await persistResults(tenantId, status.items);
      completed = true;
    }
  }
}, { connection: redis });
```

**Mapping table (PTF → Nemovizor):**
- PTF `offer_type` → Nemovizor `listing_type` (prodej/pronajem → sale/rent)
- PTF `property_type` → Nemovizor `category` (byt/dum/pozemek → apartment/house/land)
- PTF `disposition` → Nemovizor `subtype` (2+kk → "2+kk")
- PTF `price` + `currency` → `price` + `price_currency`
- PTF `photos` array → `images: [{ url, order }]`
- PTF `agent_id` → `broker_external_id: "ptf-{agent_id}"`

## 8. ZÁVĚR

Nemovizor MVP je **velmi dobře navržená platforma s pevnými základy**. Architektura je moderní, kód je čistý, a unikátní AI-first design (MCP server, OpenAPI 3.1, Zod schemas) tě staví **napřed před konkurenci** (Sreality, Bezrealitky) o 1-2 roky v oblasti agent-readiness.

**Hlavní riziko:** Security díry (RLS, Stripe webhook) které by při launch mohly vést k data corruption nebo finančnímu zneužití.

**Hlavní příležitost:** First-mover v AI-first realitním marketplace — rychlá implementace MCP rozšíření + PTF integrace = unikátní pozice před konkurencí.

**Doporučení:** Soustředit Sprint 1A (security) jako absolutní priority, paralelně rozjet PTF integraci jako proof-of-concept Import API. Po stabilizaci přejít na AI-first expanze (Fáze 2).
